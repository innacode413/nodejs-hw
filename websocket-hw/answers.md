# Відповіді на питання (WebSocket чат)
# Студентка: Інна

## 1. Чим цей чат принципово відрізняється від чату, зробленого на звичайних HTTP-запитах? Чому HTTP погано підходить для такої задачі?

У HTTP клієнт сам ініціює запит, і сервер може лише відповісти. Щоб отримувати нові повідомлення в реальному часі, клієнт мав би постійно опитувати сервер (polling) — це навантажує і сервер, і мережу. WebSocket тримає постійне з'єднання, і сервер може сам надсилати дані клієнту в будь-який момент без додаткових запитів. HTTP створювався для "запит-відповідь", а не для реалтайму.

## 2. Що означає схема ws:// і чим відрізнялась би wss://?

ws:// — це звичайне (незашифроване) WebSocket-з'єднання, як http://. wss:// — це захищене WebSocket-з'єднання через TLS, як https://. wss шифрує всі дані, тому його варто використовувати в продакшені, особливо якщо передаються особисті дані.

## 3. Що треба було б змінити, щоб надсилати повідомлення лише одному конкретному користувачу?

Потрібно ідентифікувати користувачів — наприклад, при підключенні зберігати socket у мапі за унікальним іменем або ID. Тоді замість `wss.clients.forEach` знаходити потрібний socket у мапі й надсилати лише йому.

## 4. Чому в завданні 5 час додається на сервері, а не на клієнті? Який у цьому сенс?

Якщо додавати час на клієнті, то в кожного користувача буде свій локальний час, який може відрізнятися (різні часові пояси, неточний годинник). Сервер додає єдиний "офіційний" час для всіх, щоб усі бачили однаковий час відправлення. Крім того, сервер бачить момент отримання повідомлення найточніше.
